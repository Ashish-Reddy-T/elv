## Key Decisions

1. **DINOv2 as SVA query base** (not SigLIP): DINOv2's 1369 tokens have 1:1 correspondence with 3D positions from depth backprojection. Using DINOv2 as query base eliminates position pooling entirely.

2. **e^(1/3) frequency ratio**: GridPE paper (Li et al. AAAI 2025) proves r=e^(1/p) minimizes grid cells for p-dimensional space. For p=3: e^(1/3)~1.396. The golden ratio (~1.618) was accidentally near-optimal for 2D (e^(1/2)~1.649).

3. **96→128 padding with identity pairs**: Qwen3-VL expects head_dim=128. Our 96 icosahedral dims are padded with 16 pairs of (cos=1, sin=0), making those dims position-agnostic for spatial tokens. Text tokens still use all 128 dims via M-RoPE.

4. **DeepStack over gated cross-attention**: Removes ~378M trainable params. DeepStack performs `hidden_states[mask] += visual_embeds` at early LLM layers — same function as gated cross-attn but with zero additional parameters.

5. **Graceful rope patch failure**: `apply_rope_patch` is called in `_initialize_loaded_model` but wrapped in try/except for mock models in tests that don't have the full Qwen3-VL structure.

## Training Pipeline Critique: Benchmarks, Datasets, and Ablations

### 1. Assessment of Current Planned Benchmarks

#### Well-Suited Benchmarks (Strong Choices)

**VSI-Bench** is the single best headline benchmark for this project. The 40-point gap between VLMs and humans on metric spatial tasks (distance estimation, room size) is precisely what GATr + IcosahedralRoPE3D is designed to close. All underlying scenes (ARKitScenes, ScanNet, ScanNet++) have GT 3D reconstructions, so depth can be rendered at 518x518 for the geometric branch.

**SQA3D** is equally well-matched. Situated QA with agent position and orientation directly exercises IcosahedralRoPE3D's ability to encode the agent's 3D viewpoint. The question-type breakdown (what/where/how many/spatial relation) isolates which spatial abilities improve. Human ceiling ~90% vs. SOTA ~55% leaves substantial room.

**VLN-CE (R2R, RxR)** is the community-standard navigation benchmark and must be included. However, VLN-CE SOTA systems (V2-VLNCE ~72% SR) use panoramic views, waypoint prediction, and BEV representations. A VLM-based agent on single egocentric frames has a structural disadvantage. **The more informative metric is SPL (path quality)**, not raw SR — if the model understands 3D geometry, it should take fewer wasted steps.

**ObjectNav HM3D** tests exploration efficiency. Report SoftSPL alongside SR, as it captures whether the agent moves efficiently even when it fails.

**NavTrust** is an excellent diagnostic. If the architecture thesis is correct — GATr operates on depth geometry rather than pixel appearance — robustness under visual corruptions should be a natural advantage.

#### Fit Concern

**VLN-CE RxR** multilingual instructions (Hindi, Telugu) introduce a confound. Recommend reporting English-only subset for clean ablation.

### 2. Missing Benchmarks (Recommended Additions)

#### High Priority — Promote to P0/P1

| Benchmark | Why | Data Source |
|---|---|---|
| **Spartun3D** | Situated captioning with agent orientation in ScanNet — directly tests IcosahedralRoPE3D encoding of viewpoint. 133K examples. Complements SQA3D's multiple-choice with free-form generation. | ScanNet |
| **EmbodiedScan** (CVPR 2024) | 5K+ scenes, 1M+ language descriptions. 3D visual grounding, 3D visual QA, 3D dense captioning. Built on ScanNet, 3RScan, Matterport3D with full GT depth. Most comprehensive embodied 3D understanding benchmark available. | ScanNet + 3RScan + MP3D |
| **Scan2Cap** (CVPR 2021) | Dense captioning in 3D. Tests whether GATr's equivariant representations enable precise spatial language ("the chair is 1.5m to the left" vs. "the chair is near the desk"). CIDEr on spatial descriptions directly measures spatial language quality. | ScanNet |
| **MP3D-EQA** | Navigation to answer spatial questions. Bridges static QA (SQA3D) and navigation (VLN-CE). Tests whether spatial understanding guides exploration toward answering a question. | Matterport3D |

#### Custom Equivariance Tests (No Existing Benchmark)

These would constitute the strongest evidence for IcosahedralRoPE3D:

1. **Rotation test**: Rotate input depth map by 90/180/270° — check whether spatial relation predictions rotate correspondingly. IcosahedralRoPE3D should handle this; M-RoPE should fail.
2. **Translation test**: Shift agent position by known offset — check whether distance estimates update correctly. Tests metric (not just ordinal) distance encoding.
3. **Scale test**: Scale all depths by 2× — check whether distance estimates double. GATr's equivariance should preserve this; standard MLP on depth would not.

#### Deprioritize

- **REVERIE**: Navigation + remote grounding, but grounding depends on object detection quality more than spatial reasoning. Mid-30s SR ceiling = noisy results.
- **SITE meta-benchmark**: Mixed GT-depth/no-GT-depth makes results hard to interpret.

### 3. Ablation Design Critique

#### Icosahedral (6 dirs) vs. Tetrahedral (4 dirs) — H2f

Well-motivated. The mathematical argument is clean (covering number theory on S²). However:

**Concern**: The switch from 64 dims (4×8×2, matching Qwen3-VL's 64 rotary pairs exactly) to 96 dims (6×8×2, requiring 16 identity-padded pairs) introduces structural asymmetry. The 16 identity pairs contribute no positional information — they are deadweight dimensions where spatial tokens get no positional signal while text tokens still get M-RoPE. This could hurt cross-modal attention for those dim-pairs.

**Recommendation**: Add a third condition — 6 icosahedral directions × 6 frequencies = 72 dims, padded with 28 identity pairs. If 96 > 72 > 64, the icosahedral benefit is real. If 96 ≈ 72 ≈ 64, simplify.

#### DeepStack vs. Gated Cross-Attention

The comparison is worth making, but note the asymmetry: DeepStack has zero additional trainable parameters while gated cross-attention adds 377M. Need parameter-matched comparison — e.g., take those 377M params and put them into larger LoRA rank or more SVA layers instead.

#### DINOv2 Query Base (1369) vs. SigLIP Query Base (576)

**Missing ablation**: What if DINOv2 tokens (1369) are the query base and SigLIP tokens (576) are upsampled for KV? This preserves DINOv2's fine-grained spatial structure rather than compressing through cross-attention.

**Also missing**: Purely learnable SVA queries (not initialized from any encoder). This isolates whether SigLIP/DINOv2 initialization actually helps.

### 4. Training Pipeline Critique

#### Pre-alignment

**Concern**: LLaVA-558K is generic image-caption data. Pre-alignment on "a dog sitting on a couch" may not teach projectors to preserve spatial information. Consider supplementing with spatially-rich captions — ScanNet dense captions, ScanRefer descriptions, or synthetic spatial descriptions from Habitat ("The kitchen counter is 2.3m ahead and slightly to the left").

**Concern**: GATr projector pre-alignment on LLaVA-558K is questionable. GATr produces geometric features from 3D point clouds. LLaVA-558K has no depth/3D. During pre-alignment, GATr receives zero/random inputs. Either skip GATr pre-alignment or pre-align using Habitat scenes with GT depth and synthetic spatial captions.

#### SFT

**Concern**: SFT data is overwhelmingly navigation-focused. SQA3D is the only non-navigation spatial dataset. Yet two of four P0 benchmarks (VSI-Bench, SQA3D) are static spatial QA. The model will be undertrained for its most important evaluation tasks.

**Recommended additions for SFT**: ScanRefer (36K), ScanQA (41K), Nr3D (83K), Sr3D (56K).

**Concern**: ScaleVLN's 4.9M synthetic instructions will dominate SFT without careful mixing. Cap at 200–500K samples.

#### GRPO

**Strength**: Five-term reward (format, progress, collision, goal, consistency) with lexicographic curriculum gating is well-motivated.

**Concern**: GRPO with G=8 requires 8 forward passes per prompt. At ~65ms/step, ~800 tokens, batch=32: each GRPO update takes ~17s of generation time alone. Over 6 epochs × 50K prompts = ~3,400 GPU-hours for generation — already exceeding the 1,200 H100-hours budgeted. Either reduce G to 4 (or G=2 per "It Takes Two" paper), reduce prompt count, or revise budget.

**Concern**: The curriculum schedule (6 epochs, piecewise-linear) is rigid. Consider milestone-based transitions: advance when format accuracy >95% on held-out set, regardless of epoch count.

**Missing**: How are reference policy logprobs computed for the KL penalty? If using the SFT checkpoint as reference, it must stay frozen in memory alongside the training model — roughly doubling VRAM.

#### fDPO

**Concern**: fDPO requires paired preference data. Your pipeline generates GRPO rollouts (scalar rewards) but not explicit preference pairs. Need a data generation step: for each prompt, take highest/lowest reward completions from G=8 as chosen/rejected.

**Concern**: The segment mask (grounding vs. reasoning) requires parsing model outputs. If the model hasn't learned clean formatting yet (early training), the mask is noisy. fDPO should only apply after GRPO-quality format compliance — complicating the comparison.

### 5. Dataset Diversity Assessment

| Domain | Current Data | Samples | Gap |
|---|---|---|---|
| Pre-alignment | LLaVA-558K | ~50K subset | Generic captions, no spatial focus |
| Navigation SFT | R2R-CE, RxR-CE, ScaleVLN | ~5M total | Overwhelmingly navigation-only |
| Spatial QA SFT | SQA3D | 33K | Single dataset, single task format |
| GRPO | Habitat rollouts | Generated | Only navigation episodes |

**Critical gap**: Static spatial reasoning data is severely underrepresented. SQA3D is the only non-navigation spatial dataset in SFT, yet two P0 benchmarks test static spatial QA.

**Scene diversity concern**: ScanNet (~1,500 scenes), Matterport3D (~90), HM3D (~800) overlap heavily across benchmarks. SFT on ScanNet-derived data + eval on ScanNet-derived benchmarks risks memorizing scene layouts. Enforce strict scene-level train/test splits.

**Scale concern**: Total unique spatial data (excluding ScaleVLN synthetic) is ~180K samples. With 206M trainable params, SFT overfitting is a real risk. Monitor val loss closely, use early stopping.

### 7. Additional Findings (Second Research Pass)

**Critical observation**: None of the current 6 benchmarks cleanly isolate the geometry branch's contribution. The permutation test (H3c) tests whether spatial token *ordering* matters, but not whether *3D geometric features* matter specifically. Need a benchmark where full model dramatically outperforms the no-GATr ablation.

**Strongest "smoking gun" candidate: 3DSRBench (HSSD subset)**
- GT depth available (HSSD is Habitat-compatible)
- 12 fine-grained 3D reasoning question types: height, location, orientation, multi-object
- Manual annotations ensuring questions genuinely require 3D understanding
- Synthetic subset testing uncommon viewpoints (where implicit spatial priors fail)

**Additional benchmarks to consider:**
- **SpatialBench** (2024-2025): Unifies metric depth estimation, proximity, contact/reaching, counting, size reasoning. Directly tests whether GATr + IcosahedralRoPE3D enables *metric* spatial reasoning.
- **OpenEQA** (Meta, CVPR 2024): 1600+ questions, 180+ environments. Episodic-memory mode compatible with our pipeline. Best VLM ~50% vs human 87%.
- **MindCube** (ICCV 2025): 21K questions testing spatial mental models, perspective-taking, mental simulation. Existing VLMs show near-random performance.

**Metric spatial reasoning data gap** (most significant dataset gap):
Current training data is entirely qualitative spatial reasoning. No training data teaches the model to output metric distances, sizes, or angles. Without explicit metric supervision, GATr may improve qualitative reasoning but the model will never articulate metric spatial knowledge. Recommendation: generate 10K-50K synthetic metric QA from Habitat GT depth ("How far is X from Y?" with numeric GT answers).

**Additional ablation sweeps to add:**
- SVA layer count (1/2/4) — validates major architectural choice, ~86M params at stake
- GATr block count (4/8/16) — highest-risk module, convergence concerns from hPGA-DP
- LoRA rank sweep (8/16/32/64) — validates compute allocation
- GRPO group size (2/4/8) — "It Takes Two" paper shows G=2 matches G=16, potential 70%+ training time savings
